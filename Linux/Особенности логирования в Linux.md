Логирование в операционных системах Linux играет ключевую роль для отслеживания действий злоумышленника. В этом шаге мы рассмотрим особенности логирования в ОС Linux, а также рассмотрим основные службы, отвечающие за логирование.

## Основные файлы журналов в Linux-системах

Как правило, файлы журналов представляют из себя обычные текстовые файлы, где каждая строчка описывает одно конкретное событие. Обычно в этой строчке можно найти отметку времени (день, час, минуты, и часовой пояс), тип события (ошибка, предупреждение, информационное событие) и краткое описание самого события. Файлы журналов как правило принадлежат пользователю root и находятся в каталоге `/var/log`.  В этой директории мы можем найти следующие файлы журналов:

- `rsyslog/journald` — логи, генерируемые менеджером;
- `kernel.log` — логи ядра ОС;
- `audit.log` — логи ядра, которые пишет auditd, если он настроен;
- `auth.log/secure` — журнал событий безопасности;
- Логи приложений — например, веб-севера, БД или прочих. 

> Поскольку дистрибутивов Linux великое множество, файлы журналов не всегда имеют одинаковые названия и локации в разных ОС. Например, логи событий для команд `sshd`, `sudo` и прочих действий, связанных с безопасностью в Ubuntu, находятся в `/var/log/auth.log`, а в CentOS в `/var/log/secure`. Также, файлы менеджера журналирования находятся в `/var/log/syslog` для Ubuntu, а в CentOS - `/var/log/messages`. 

## Daemon (демоны) в системах Linux

На большинстве машин Linux, подключенных к корпоративной среде, как правило, работают какие-либо важные приложения, например веб-сервер или база данных. Одной из основных особенностей ОС Linux является использование демонов — фоновых программ, которые могут работать без взаимодействия с пользователем. Рассмотрим демон Nginx, отвечающий за работу веб-сервера. Любой пользователь может проверить статус демона используя команду `systemctl`. 

![](https://ucarecdn.stepik.net/982b7d90-2381-4fcf-a988-f42d6469531e/)

Как мы видим на картинке выше, сервис (демон) активен и работает. Для остановки или перезапуска сервиса требуются права суперпользователя. Ubuntu и CentOS используют команду `service`, которая является оболочкой для systemctl.  

## Демон Syslog

**Syslog-демон** — это менеджер журналов, появившийся более 40 лет назад в дистрибутивах BSD и до сих пор остающийся основным решением для управления логами. Основная задача демона Syslog — прием событий от локальных служб и последующая обработка и запись их в файлы журналов `/var/log`, либо дальнейшая пересылка событий по сети.

По факту, существует три вида Syslog:

- Syslog как менеджер журналов;
- Syslog как сетевой протокол передачи данных;
- Syslog как формат предоставления данных.

Сегодня помимо Syslog также можно встретить Nxlog и Syslog-NG в качестве альтернатив, но поскольку Syslog преобладает на рынке, мы будем рассматривать его.

Обычно приложения отправляют события менеджеру журналов в своем собственном формате, и для удобства чтения логов из разных источников эти сообщения было бы хорошо стандартизировать. 

Возьмем событие `sshd` и попробуем поменять его формат.

![](https://ucarecdn.stepik.net/42c2a2d8-eb77-49ea-b7c2-c992845483a1/)

_Формат сообщений демона `sshd` по умолчанию._

Для преобразования событий в стандартизированный формат (мы будем использовать [RFC3164](https://datatracker.ietf.org/doc/html/rfc3164)) нам нужно создать шаблон в `/etc/rsyslog.d/50-default.conf`. Пример первых строк файла, тут мы меняем формат логов для `auth,authpriv.*`:
```
#  Default rules for rsyslog.
#
#                       For more information see rsyslog.conf(5) and /etc/rsysl>
$template RFC3164fmt,"<%PRI%>%TIMESTAMP% %HOSTNAME% %syslogtag%%msg%\n"
#
# First some standard log files.  Log by facility.
#
auth,authpriv.*                 /var/log/auth.log;RFC3164fmt
*.*;auth,authpriv.none          -/var/log/syslog
#cron.*                         /var/log/cron.log
#daemon.*                       -/var/log/daemon.log
kern.*                          -/var/log/kern.log
#lpr.*                          -/var/log/lpr.log
mail.*                          -/var/log/mail.log
#user.*                         -/var/log/user.log

```

_Пример шаблона формата сообщений_

После изменения конфигурации нам необходимо перезагрузить службу `rsyslog`, и, как мы видим, к нашему событию добавилось значение приоритета. 
```
<38>Mar 16 16:58:42 ubuntuWS sshd[3057]: Accepted password for serj from 192.168.122.144 port 34678 ssh2
```

Давайте разберем данное событие:

- `<38>` — значение приоритета, получается за счет комбинирования значений `facility` и `severity`. Чем меньше цифра, тем важнее событие;
- `Mar 16 16:58:42` — временная метка;
- `ubuntuWS` — имя сервера, на котором произошло событие: поле важно при отправке логов на другой сервер;
- `sshd` — имя сервиса, который создал запись о событии;
- `[3057]` — номер процесса, Process ID;
- `Accepted password....` — собственно, само событие: SIEM в процессе нормализации логов выделит дополнительные поля, такие как `source_ip`, `action`, `user` и тд..

Значение приоритета syslog считается по формуле `facility * 8 + severity`. `Facility` имеет заданные значения, и определяет к какому типу относится событие. Например, `facility #0` относится к событиям ядра, а `#4` - к событиям систем безопасности.

![](https://ucarecdn.stepik.net/cfbadffe-292d-4b78-9c22-f1a600292e29/)

`Severity` отвечает за уровень критичности событий, где 0 это максимум(emergency), а уровень 7 - малоинформативные данные, используемые для отладки(debug).

В нашем примере число 38 получается за счет `facility` 4 и `severity` 6 (info). 

## Journald

Современные Linux-системы помимо rsyslog параллельно используют journald для обработки и хранения логов. 

Journald является частью systemd, менеджера конфигурации и управления службами (демонами), который после анонса в 2015 году стал де-факто стандартом для большинства Linux-систем. Systemd_journald или просто journal отвечает за централизованное управление логами. Для обеспечения целостности данных journald хранит логи в бинарном формате, в отличие от rsyslog, который использует обычные текстовые файлы.  Предполагается, что journald в будущем заменит rsyslog, который сам в свою очередь заменил более старую версию syslog. Для обеспечения обратной совместимости на большинстве Linux систем rsyslog работает вместе с journald, получая от него сообщения и записывая их в файлы. Чтобы убедиться в этом, давайте посмотрим, какой процесс записывает события в файл журнала с помощью команды `lsof`.

![](https://ucarecdn.stepik.net/1d1854f8-5f15-45a6-8770-0cbdf04f9fc9/)

Обратите внимание, что параллельно со старым модулем imuxsock в качестве источника событий используется модуль imjournal. В этом случае каждое событие дублируется — одна копия записывается в текстовый файл демоном rsyslog, а вторая копия помещается в журнал journald.
